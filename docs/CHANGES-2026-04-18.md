# Healix v1.1 — Pipeline Error Surfacing Hardening (2026-04-18)

This note explains a batch of changes landed on 2026-04-18 to fix a real
failure hit against **pm-app** and to make sure the same class of bug can
never render as `stage: unknown, reason: unknown_reason` on the dashboard
again. Review this alongside the diffs; file-level rationales below.

## The bug, in one screenshot

Pipeline failed with:

```
Playwright execution failed with exit code 1: SyntaxError:
The requested module './__healix-fixture' does not provide an export
named 'expect' | SyntaxError: ... | Error: No tests found
```

Dashboard rendered:

- Banner: `stage: unknown, reason: unknown_reason` (useless to the user).
- Stats strip: `Total 1, Failed 1` (lie — zero tests actually ran).
- A fake "Pipeline" test row with 0.00s duration.

Three independent defects stacked up:

1. **Fixture ESM/CJS mismatch.** pm-app declares `"type": "module"` in its
   `package.json`. Healix wrote `__healix-fixture.js` using CJS
   `module.exports = { test, expect, request }`. Node treats a `.js` in an
   ESM package as ESM; ESM named imports can't synthesize from
   `module.exports` → `expect` is not found → Playwright loader crashes →
   "No tests found".
2. **`stage/reason` fell through to "unknown"** because
   `PlaywrightIntegration.runTests` throws a plain `Error` without
   `error.diagnostics`, and the catch block that builds `pipelineError`
   defaulted both fields when nothing more specific was attached.
3. **Synthetic pipeline-error row was double-counted** into
   `testRun.total_tests` / `failed_tests`, so the KPI strip lit up with
   `1 / 1` when zero real tests executed.

## What changed, by file

### `testbot-mcp/src/pipeline-worker.js` (modified)

- **`detectProjectModuleType(projectPath)`** — reads target project
  `package.json`, returns `'module'` when `type === 'module'`, otherwise
  `'commonjs'`. Missing/invalid `package.json` → `'commonjs'` (safe
  default matching Node).
- **`getCursorFixtureContent(serializedInitScript, moduleType)`** — body
  of `__healix-fixture.js` now keyed to project module type:
  - `'module'` → `import { test as base, expect, request } from '@playwright/test'; ... export { test, expect, request };`
  - `'commonjs'` → existing `const { ... } = require(...); module.exports = { ... };`
  - TS body (`__healix-fixture.ts`) unchanged — Playwright's TS loader
    owns it regardless of project type.
- **`ensureCursorFixtureFiles(generatedDir, projectPath)`** — now takes
  the project path and dispatches to the right body. Single call site
  updated to pass `projectPath`.
- **`buildPipelineDiagnostics` + the outer catch-block pipelineError
  synthesis** — both now call `classifyPipelineErrorFromStderr` and merge
  its `{stage, reason, errorCode, userFacingMessage}` in. Dashboard never
  sees `'unknown'` or `'unknown_reason'` from these code paths.
- Synthetic pipeline-error result rows are tagged `isPipelineSynthetic:
  true` so ingest/dashboard can distinguish them from real failures.
- Exports added: `detectProjectModuleType`, `getCursorFixtureContent`,
  `ensureCursorFixtureFiles` (for unit tests).

### `testbot-mcp/src/failure-triage/pipeline-error-classifier.js` (new)

Deterministic stderr/stdout pattern classifier, modelled after the T3
runtime classifier but scoped to **pre-test pipeline failures**. First
match wins; rules are ordered so specific patterns beat generic ones
(e.g. `fixture_module_type_mismatch` runs before `generated_test_syntax_error`).

Rules currently shipped:

| id | stage | errorCode |
|---|---|---|
| `fixture_module_type_mismatch` | execution | `FIXTURE_MODULE_TYPE_MISMATCH` |
| `generated_test_syntax_error` | execution | `GENERATED_TEST_SYNTAX_ERROR` |
| `no_tests_loaded` | execution | `NO_TESTS_LOADED` |
| `missing_playwright_dependency` | validation | `PLAYWRIGHT_DEPENDENCY_MISSING` |
| `missing_node_module` | validation | `MISSING_DEPENDENCY` |
| `server_unreachable` | server_start | `SERVER_START_TIMEOUT` |
| `playwright_list_failed` | validation | `GENERATION_VALIDATION_FAILED` |
| `expo_dependency_validation` | server_start | `EXPO_DEPENDENCY_VALIDATION_FAILED` |
| `time_budget_exceeded` | execution | `TIME_BUDGET_EXCEEDED` |

Fallback emits `id: 'unclassified_pipeline_error'` with `stage: 'execution'`
and a usable user-facing message. **Never** returns `'unknown'`.

`mergeWithClassification(partial, classification)` — call-site helper
used by `buildPipelineDiagnostics`: explicit caller fields win, but
placeholder `'unknown'` / `'unknown_reason'` are replaced.

**Extending this:** add an entry to `CLASSIFIERS` plus a test in
`test/pipeline-error-classifier.test.js`. Keep regexes tight — over-broad
patterns mask finer-grained rules below them.

### `webapp/src/app/(dashboard)/test-run/[id]/page.tsx` (modified)

- KPI strip (`totalTests`, `passedTests`, `failedTests`, `skippedTests`,
  `passRate`) is zeroed out when `pipelineError` is set. The synthetic
  row no longer inflates the counts.
- Pipeline-error banner (from phase T1) remains the single source of
  truth for "something failed before tests could run".

## Why this matters for reviewers

- The fixture emitter is now **aware of target-project module type**.
  Any future change to the fixture must preserve both code paths
  (ESM + CJS) and keep the TS body untouched. The regression test
  guards this.
- `stage/reason = unknown` is now a **bug**, not a valid state. If the
  dashboard ever shows it, the classifier is missing a rule — add one,
  don't paper over it.
- The KPI strip truth rule is: **if `pipelineError` is set, no tests
  ran. Stats are zero. The banner is the message.**

## Agent feedback loop: `healix_check_run_status` (landed in this batch)

The screenshot also exposed a second-order problem: **the Cursor agent
went idle after kicking off the run.** `healix_test_my_app` returns
immediately with a `configUrl` so the user can submit the form, but
there was no mechanism for the agent to find out whether the run
eventually succeeded, errored, or produced pipeline-level diagnostics —
so it went silent while the background pipeline was still executing.

Fixed by adding a new MCP tool:

- **`healix_check_run_status({ runId, projectPath? })`** — reads
  `healix-reports/.runs/{runId}/status.json`. Returns a JSON payload
  with `phase`, `isTerminal`, `errorCode`, `dashboardUrl`, `configUrl`,
  and an `agentInstructions` field that explicitly tells the Cursor
  agent what to do next.
- For **non-terminal phases**: `isTerminal: false` + "call
  `healix_check_run_status` again in ~15 seconds. Do not conclude the
  task until isTerminal is true."
- For **terminal phases** (`completed` / `error` / `error_reported` /
  `failed`): `isTerminal: true` + the full T7 action plan from
  `buildAgentResponse` (same structure `healix_analyze_failures`
  returns: `auto_apply`, `surface_for_approval`, `app_regressions`,
  `environment_issues`, `pipeline_error`). If no report is on disk
  yet (e.g. pipeline errored before any tests ran), the payload falls
  back to message + errorCode + dashboardUrl with instructions to
  relay them to the user.
- **Resilience**: corrupt/mid-write `status.json` → `isTerminal: false`
  with `phase: 'transient_read_error'` and a "try again in ~5s"
  instruction. Never throws.

`healix_test_my_app`'s final response was also rewritten. It now
includes an `agentInstructions` field spelling out, in natural language
the model will obey: *"The test pipeline is running in the BACKGROUND.
You MUST now call `healix_check_run_status` with { runId: "…" } every
~15 seconds until `isTerminal` is true. Do NOT hand control back to
the user, summarize, or declare the task done while `isTerminal` is
false."*

Tests (`testbot-mcp/test/check-run-status.test.js`, 7 cases):

- Missing `runId` → clean error.
- Unknown `runId` → `RUN_NOT_FOUND`.
- Running phase → `isTerminal: false` + "poll again in ~15s".
- `awaiting_config_ui` → `isTerminal: false` + configUrl surfaced.
- `completed` without report → `isTerminal: true` + share-dashboard.
- `error` phase → `isTerminal: true` + errorCode surfaced.
- Corrupt status.json → `transient_read_error`, never throws.

**Total suite: 98 tests, 97 pass, 1 skipped** (the
`vm.SourceTextModule` ESM-parse test is skipped unless Node is
launched with `--experimental-vm-modules`; the static-regex checks
above it already guard the pm-app regression).

## Token/credits display consolidation

A second regression surfaced in the same session: the dashboard was
showing **two different "quota" numbers** that never agreed. The
sidebar said "1032 tokens" (green), and the home-page "My Plan" card
said "50/50 used" (capped). Root cause was two independent quota
systems coexisting on `profiles`:

| Column                 | Who reads it                                         | Who writes it |
| ---------------------- | ---------------------------------------------------- | ------------- |
| `tokens_remaining`     | `/api/generate-tests`, `/api/parse-prd`, `/api/analyze-failures` | deducted on every AI call |
| `credits_remaining`    | `/api/mcp-auth/validate`, home "My Plan" card        | never decremented by the runtime |

So `credits_remaining` rotted at its default, while `tokens_remaining`
tracked real usage. The sidebar read tokens, the plan card read
credits, and the MCP auth gate blocked on credits while actually
billing tokens. Stale/made-up defaults (`?? 240000`, `?? 130 / ?? 500`,
Sidebar `50/50`) compounded the confusion.

Fix in this batch:

- **`webapp/src/lib/tokens.ts`** — single source of truth:
  `REAL_TOKENS_PER_DISPLAY_UNIT = 4_800` constant + `toDisplayUnits()`
  helper. Exported for every surface that displays tokens.
- **`webapp/src/app/(dashboard)/layout.tsx`** — sidebar now reads
  `toDisplayUnits(profile?.tokensRemaining)` without any fake fallback.
  Missing/null tokens render as `0`, not `50`.
- **`webapp/src/components/dashboard/Sidebar.tsx`** — default props
  `0/0` instead of `50/50`, so an unloaded or out-of-quota user isn't
  misrepresented as "half-full".
- **`webapp/src/app/(dashboard)/home/page.tsx`** — "My Plan" card now
  reads `profile.tokens_remaining` / `tokens_total` via the same
  `toDisplayUnits`. The `credits_remaining` column is no longer surfaced.
- **`webapp/src/app/(dashboard)/plan-billing/page.tsx`** — imports
  `toDisplayUnits` from `@/lib/tokens` instead of maintaining its own
  `REAL_TOKENS_PER_UNIT` constant.
- **`webapp/src/app/api/mcp-auth/validate/route.ts`** — now blocks on
  `tokens_remaining <= 0` with error code `NO_TOKENS` (previously
  `NO_CREDITS` on a column the runtime never decremented). Response
  still returns `creditsRemaining` for back-compat with older MCP
  clients but also returns `tokensRemaining`.
- **`webapp/src/app/api/profile/route.ts`** — removed the misleading
  `?? 5_000_000` fallback on the token fields. If the DB row has null,
  the payload reports `0`.

Net effect: sidebar, home plan card, billing page, and auth gate all
read the same `tokens_remaining` field through the same display helper,
so the numbers on the screen cannot disagree with each other or with
billing reality.

## Verification checklist for reviewers

1. `cd testbot-mcp && node --test test/` — all green.
2. Point Healix at any repo with `"type": "module"` in `package.json`.
   Run it. Confirm `tests/generated/__healix-fixture.js` contains
   `export { test, expect, request };` and no `module.exports`.
3. Point Healix at a CJS project (no `"type"` or `"type": "commonjs"`).
   Confirm the fixture still uses `require` + `module.exports`.
4. Force a pipeline-level failure (e.g. uninstall `@playwright/test` in
   the target, or corrupt one generated spec). Dashboard banner shows a
   concrete `stage` and `reason`, never `unknown`. KPI strip shows
   `Total 0 / Failed 0`.
